## 各メンバーへの質問

以下の質問を各メンバーに `SendMessage(type: "message")` で送信し、回答を収集します。

### 質問1: 担当タスクの概要
```
今回担当したタスクを簡潔に説明してください：
- タスクID:
- タスク名:
- 実装内容:
- 所要時間の感覚: [短い/適切/長い]
```

### 質問2: 技術的な学び
```
今回のタスクで学んだ技術的な知見を教えてください：
- 新しく知ったこと:
- 理解が深まったこと:
- 今後も活用できそうなテクニック:
```

### 質問3: 遭遇した課題と解決方法
```
実装中に遭遇した課題とその解決方法を教えてください：
- 課題:
- 解決方法:
- 解決までの試行回数:
- 次回同じ課題に遭遇したら:
```

### 質問4: 他メンバーとの連携状況
```
チーム内での連携について教えてください：
- 他メンバーとのやり取り回数:
- やり取りの内容: [質問/情報共有/レビュー依頼/その他]
- スムーズだった点:
- 改善できそうな点:
```

### 質問5: 改善提案
```
プロジェクト全体の改善提案があれば教えてください：
- CLAUDE.md への追記提案:
- スキルの改善案:
- スクリプト・フックの改善案:
- permissions設定の改善案:
- その他:
```

## リーダーの分析

リーダーは以下の観点で自己分析を実施します。

### 1. タスク設計の妥当性

#### 粒度
- [ ] タスクの粒度は適切だった（大きすぎず、小さすぎず）
- [ ] 各タスクの完了条件が明確だった
- [ ] メンバーが理解しやすい説明だった

**改善点**:
- [例: タスクXはスコープが広すぎた。次回は2つに分割する]

#### 依存関係
- [ ] タスク間の依存関係が正しく設定されていた
- [ ] ブロッカーが最小限だった
- [ ] 並列作業が効率的に進んだ

**改善点**:
- [例: タスクYとZは依存関係があると思ったが、実際は並列実行可能だった]

#### 優先順位
- [ ] 重要度の高いタスクから着手できた
- [ ] クリティカルパスを意識した順序付けができた

## Issue化判断チェックリスト

各改善点について、以下のチェックリストで Issue化の必要性を判断します。

### 必須条件（すべて満たす必要がある）
- [ ] 複数回の再現性がある（or 今後も発生する可能性が高い）
- [ ] プロジェクト全体の効率向上に寄与する
- [ ] セキュリティ・品質に関わる（or 重要度が高い）

### 除外条件（1つでも該当したらIssue化しない）
- [ ] 一時的な問題（今回限りの特殊事情）
- [ ] 小さな修正で対応可能（5分以内で直接実施できる）
- [ ] 再利用性がない（このプロジェクト専用で他に適用できない）

### Issue化する改善点のリスト
1. [改善点のタイトル] - カテゴリ: [A/B/C/D]
2. [改善点のタイトル] - カテゴリ: [A/B/C/D]
